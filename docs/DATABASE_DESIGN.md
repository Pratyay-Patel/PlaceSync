## 1. Overview

PlaceSync uses a single PostgreSQL 15+ database instance (hosted on Supabase in V1) shared by all modules of the modular monolith. The schema is divided into 18 tables across 13 functional domains. All DDL is managed exclusively through Flyway versioned migration scripts — no manual DDL against the production database is permitted.

The schema is designed for:

- **Correctness** — FK constraints, CHECK constraints, and UNIQUE constraints enforce data integrity at the database level, independent of application code.
- **Security** — Sensitive columns (password hashes, token hashes) are never exposed in views or denormalized outside their owning tables.
- **Performance** — Indexes are defined on every column used in WHERE clauses, JOIN conditions, and ORDER BY expressions identified from the SRS requirements.
- **Auditability** — An append-only `audit_log` table captures all write operations with before/after JSONB snapshots.
- **Recoverability** — Soft delete via `deleted_at` on mutable business entities preserves data history.

## 2. Design Principles

| Principle | Implementation |
|---|---|
| **UUID primary keys** | All tables use `UUID` PKs generated by `gen_random_uuid()`. This avoids sequential ID enumeration attacks and is safe for future distributed systems. |
| **TIMESTAMPTZ over TIMESTAMP** | All timestamp columns use `TIMESTAMPTZ` (timestamp with time zone). PostgreSQL stores these in UTC internally, preventing daylight saving time bugs and satisfying the SRS requirement that all timestamps be in UTC. |
| **NOT NULL by default** | Columns are `NOT NULL` unless there is a specific business reason for nullability. Nullable columns are documented explicitly. |
| **No application-managed sequencing** | No business logic depends on the order of UUID values. Ordering always uses `created_at`. |
| **Denormalization only for auditability** | The `audit_log` table deliberately denormalizes `actor_role` and `actor_email` because those values must be captured as they were at the time of the event, not as they are now. |
| **PostgreSQL-native features** | ENUMs for type safety, `JSONB` for flexible payloads, `INET` for IP addresses, partial indexes, and `CHECK` constraints are used where appropriate. |
| **Immutable audit records** | The `audit_log` table has no `updated_at` column. No UPDATE or DELETE is permitted on it at the application layer (enforced by convention and future row-level security). |

## 3. Custom Types & Enumerations

All custom ENUM types are created in migration `V001`. Using PostgreSQL native enums instead of `VARCHAR` enforces valid values at the database level and documents the type domain directly in the schema.

| EmailVerificationToken | `email_verification_tokens` | auth | Time-limited tokens sent to new users to verify their email address. |
| PasswordResetToken | `password_reset_tokens` | auth | Time-limited tokens for password reset flows. |
| AuditLog | `audit_log` | audit | Immutable, append-only record of all write operations across the system. |

## 6. Table Schemas

The schemas below are written in PostgreSQL DDL and represent the authoritative definition of each table. These DDL statements are split across individual Flyway scripts (see Section 13 for migration ordering).

### 6.1 `users`

**Purpose:** The central identity table for every person in the system, regardless of role. A single row exists per user. Authentication credentials, role assignment, and account lifecycle state all live here. Role-specific data (profile fields, skills, company association) lives in separate tables that reference this one.

The design avoids a "God table" by keeping only authentication and account management data here. All profile data is in child tables.

```sql
CREATE TABLE users (
    id                      UUID            NOT NULL DEFAULT gen_random_uuid(),
    email                   VARCHAR(255)    NOT NULL,
    password_hash           VARCHAR(255)    NULL,
    -- NULL for OAuth2 users who never set a password
    role                    user_role       NOT NULL,
    is_email_verified       BOOLEAN         NOT NULL DEFAULT FALSE,
    is_active               BOOLEAN         NOT NULL DEFAULT TRUE,
    failed_login_attempts   SMALLINT        NOT NULL DEFAULT 0,
    locked_until            TIMESTAMPTZ     NULL,
    -- NULL = not locked; future timestamp = locked until that time
    oauth_provider          VARCHAR(50)     NULL,
    -- e.g., 'google'; NULL for email/password users
    oauth_provider_id       VARCHAR(255)    NULL,
    -- Google's 'sub' claim; NULL for email/password users
    created_at              TIMESTAMPTZ     NOT NULL DEFAULT NOW(),
    updated_at              TIMESTAMPTZ     NOT NULL DEFAULT NOW(),
    deleted_at              TIMESTAMPTZ     NULL,
    -- NULL = active; NOT NULL = soft-deleted

    CONSTRAINT pk_users PRIMARY KEY (id),

    CONSTRAINT uq_users_email
        UNIQUE (email),

    CONSTRAINT uq_users_oauth
        UNIQUE (oauth_provider, oauth_provider_id),

    CONSTRAINT chk_users_password_or_oauth
        CHECK (
            (password_hash IS NOT NULL AND oauth_provider IS NULL)
            OR
            (password_hash IS NULL AND oauth_provider IS NOT NULL)
            OR
            -- Account can have both after linking; covered by application logic
            (password_hash IS NOT NULL AND oauth_provider IS NOT NULL)
        ),

    CONSTRAINT chk_users_failed_attempts
        CHECK (failed_login_attempts >= 0),

    CONSTRAINT chk_users_oauth_consistency
        CHECK (
            (oauth_provider IS NULL) = (oauth_provider_id IS NULL)
        )
);

COMMENT ON TABLE users IS 'Central identity record for all user roles. Single source of truth for authentication.';
COMMENT ON COLUMN users.password_hash IS 'BCrypt hash (cost 12). NULL for pure OAuth2 users.';
COMMENT ON COLUMN users.oauth_provider_id IS 'Provider-issued user ID (e.g. Google sub claim). Never the provider access token.';
COMMENT ON COLUMN users.locked_until IS 'Account lock expiry after repeated failed login attempts. NULL means account is not locked.';
COMMENT ON COLUMN users.deleted_at IS 'Soft delete timestamp. NULL means active. Physical deletion does not occur in V1.';
```

### 6.2 `student_profiles`

**Purpose:** Extends the `users` table with placement-specific profile data for students. The 1:1 relationship is enforced by the UNIQUE constraint on `user_id`. Profile data is separated from auth data to keep the `users` table focused and to allow independent caching of profile content.

```sql
CREATE TABLE student_profiles (
    id                      UUID            NOT NULL DEFAULT gen_random_uuid(),
    user_id                 UUID            NOT NULL,
    first_name              VARCHAR(100)    NOT NULL,
    last_name               VARCHAR(100)    NOT NULL,
    phone                   VARCHAR(20)     NULL,
    date_of_birth           DATE            NULL,
    gender                  gender_type     NULL,
    institution             VARCHAR(255)    NOT NULL,
    department              VARCHAR(255)    NOT NULL,
    graduation_year         SMALLINT        NOT NULL,
    cgpa                    NUMERIC(3, 2)   NULL,
    -- Scale 2: e.g., 9.75. CHECK enforces valid range.
    bio                     TEXT            NULL,
    profile_picture_s3_key  VARCHAR(500)    NULL,
    -- NULL = no picture uploaded; stores S3 object key, not URL
    is_profile_public       BOOLEAN         NOT NULL DEFAULT TRUE,
    created_at              TIMESTAMPTZ     NOT NULL DEFAULT NOW(),
    updated_at              TIMESTAMPTZ     NOT NULL DEFAULT NOW(),

    CONSTRAINT pk_student_profiles PRIMARY KEY (id),

    CONSTRAINT fk_student_profiles_user
        FOREIGN KEY (user_id)
        REFERENCES users (id)
        ON DELETE RESTRICT,

    CONSTRAINT uq_student_profiles_user_id
        UNIQUE (user_id),

    CONSTRAINT chk_student_profiles_cgpa
        CHECK (cgpa IS NULL OR (cgpa >= 0.00 AND cgpa <= 10.00)),

    CONSTRAINT chk_student_profiles_graduation_year
        CHECK (graduation_year >= 2000 AND graduation_year <= 2100),

    CONSTRAINT chk_student_profiles_phone
        CHECK (phone IS NULL OR LENGTH(TRIM(phone)) >= 7)
);

COMMENT ON TABLE student_profiles IS '1:1 extension of users for students. Holds placement-relevant profile fields.';
COMMENT ON COLUMN student_profiles.profile_picture_s3_key IS 'AWS S3 object key. Access via pre-signed URL; never stored as a public URL.';
COMMENT ON COLUMN student_profiles.cgpa IS 'Current CGPA on a 0–10 scale. NULL if student has not filled this field.';
```

### 6.3 `student_skills`

**Purpose:** Stores the skills a student has listed on their profile. Modelled as a child collection (one row per skill) rather than an array column to support indexed filtering — recruiters cannot query `text[]` arrays efficiently with standard indexes.

```sql
CREATE TABLE student_skills (
    id          UUID            NOT NULL DEFAULT gen_random_uuid(),
    student_id  UUID            NOT NULL,
    skill_name  VARCHAR(100)    NOT NULL,

    CONSTRAINT pk_student_skills PRIMARY KEY (id),

    CONSTRAINT fk_student_skills_student
        FOREIGN KEY (student_id)
        REFERENCES student_profiles (id)
        ON DELETE CASCADE,

    CONSTRAINT uq_student_skills_student_skill
        UNIQUE (student_id, skill_name)
);

COMMENT ON TABLE student_skills IS 'Skills on a student profile. One row per skill to allow indexed lookups.';
```

### 6.4 `student_educations`

**Purpose:** Stores academic qualification records for a student. Multiple rows per student represent multiple degrees or certifications. Separated from the main profile because cardinality is 1:N and each entry has independent fields.

```sql
CREATE TABLE student_educations (
    id                  UUID            NOT NULL DEFAULT gen_random_uuid(),
    student_id          UUID            NOT NULL,
    degree              VARCHAR(255)    NOT NULL,
    institution         VARCHAR(255)    NOT NULL,
    field_of_study      VARCHAR(255)    NULL,
    start_year          SMALLINT        NOT NULL,
    end_year            SMALLINT        NULL,
    -- NULL if currently enrolled
    percentage_or_cgpa  NUMERIC(5, 2)   NULL,
    -- Flexible: supports both percentage (e.g., 92.50) and 10-point CGPA
    created_at          TIMESTAMPTZ     NOT NULL DEFAULT NOW(),
    updated_at          TIMESTAMPTZ     NOT NULL DEFAULT NOW(),

    CONSTRAINT pk_student_educations PRIMARY KEY (id),

    CONSTRAINT fk_student_educations_student
        FOREIGN KEY (student_id)
        REFERENCES student_profiles (id)
        ON DELETE CASCADE,

    CONSTRAINT chk_student_educations_years
        CHECK (end_year IS NULL OR end_year >= start_year),

    CONSTRAINT chk_student_educations_start_year
        CHECK (start_year >= 1980 AND start_year <= 2100),

    CONSTRAINT chk_student_educations_marks
        CHECK (percentage_or_cgpa IS NULL OR percentage_or_cgpa >= 0)
);

COMMENT ON TABLE student_educations IS 'Academic qualifications belonging to a student profile.';
```

### 6.5 `student_experiences`

**Purpose:** Stores work and internship experience history for a student. The `is_current` flag indicates an ongoing role, in which case `end_date` is NULL. This is the standard pattern for experience tracking used by most professional platforms.

```sql
CREATE TABLE student_experiences (
    id              UUID        NOT NULL DEFAULT gen_random_uuid(),
```

### 6.8 `resumes`

**Purpose:** Stores metadata for resume files uploaded by students. The actual PDF binary is stored on AWS S3; this table stores the reference (`s3_key`), display label, file size, and soft-delete status.

The `is_default` flag is constrained by a **partial unique index** — only one active (non-deleted) resume per student may have `is_default = TRUE`. This is enforced at the database level, not just the application layer.

```sql
CREATE TABLE resumes (
    id                  UUID            NOT NULL DEFAULT gen_random_uuid(),
    student_id          UUID            NOT NULL,
    label               VARCHAR(255)    NOT NULL,
    original_filename   VARCHAR(500)    NOT NULL,
    s3_key              VARCHAR(500)    NOT NULL,
    file_size_bytes     BIGINT          NOT NULL,
    is_default          BOOLEAN         NOT NULL DEFAULT FALSE,
    uploaded_at         TIMESTAMPTZ     NOT NULL DEFAULT NOW(),
    deleted_at          TIMESTAMPTZ     NULL,
    -- Soft-deleted resumes may not be selected for new applications
    -- but remain referenced by existing applications for data integrity

    CONSTRAINT pk_resumes PRIMARY KEY (id),

    CONSTRAINT fk_resumes_student
        FOREIGN KEY (student_id)
        REFERENCES student_profiles (id)
        ON DELETE RESTRICT,

    CONSTRAINT uq_resumes_s3_key
        UNIQUE (s3_key),

    CONSTRAINT chk_resumes_file_size
        CHECK (file_size_bytes > 0 AND file_size_bytes <= 10485760)
        -- Maximum 10 MB = 10 * 1024 * 1024 bytes
);

-- Partial unique index: enforces that at most one active resume per student is the default.
-- Partial indexes are a PostgreSQL feature that allows the WHERE clause,
-- preventing the constraint from applying to soft-deleted resumes.
CREATE UNIQUE INDEX uq_resumes_student_default
    ON resumes (student_id)
    WHERE is_default = TRUE AND deleted_at IS NULL;

COMMENT ON TABLE resumes IS 'Metadata for student resume files stored on AWS S3. Actual PDF bytes are not stored here.';
COMMENT ON COLUMN resumes.s3_key IS 'AWS S3 object key. Format: resumes/{studentId}/{resumeId}/{filename}. Used to generate pre-signed download URLs.';
COMMENT ON COLUMN resumes.deleted_at IS 'Soft delete. Deleted resumes remain to preserve historical application references.';
```

### 6.9 `jobs`

**Purpose:** Job postings created by verified recruiters. The `status` column drives the job lifecycle from `PENDING_APPROVAL` through `OPEN` to `CLOSED` or `EXPIRED`. Admin approval gates job visibility to students. The `deleted_at` supports admin removal of inappropriate postings without losing application history.

`compensation` is stored as `VARCHAR` rather than a numeric because it covers both full-time CTC (in lakhs, e.g., "12 LPA") and internship stipends (monthly, e.g., "₹25,000/month"), which have incompatible numeric semantics.

```sql
CREATE TABLE jobs (
    id                      UUID                NOT NULL DEFAULT gen_random_uuid(),
    recruiter_id            UUID                NOT NULL,
    company_id              UUID                NOT NULL,
    title                   VARCHAR(255)        NOT NULL,
    description             TEXT                NOT NULL,
    location_type           job_location_type   NOT NULL,
    job_type                job_type            NOT NULL,
    location_city           VARCHAR(255)        NULL,
    -- Required for ONSITE/HYBRID; NULL for REMOTE
    compensation            VARCHAR(255)        NULL,
    -- Free text: "12 LPA", "₹25,000/month", etc.
    application_deadline    TIMESTAMPTZ         NOT NULL,
    min_cgpa                NUMERIC(3, 2)       NULL,
    -- NULL = no CGPA filter; otherwise students below this threshold are blocked
    status                  job_status          NOT NULL DEFAULT 'PENDING_APPROVAL',
    approved_by             UUID                NULL,
    approved_at             TIMESTAMPTZ         NULL,
    closed_at               TIMESTAMPTZ         NULL,
    created_at              TIMESTAMPTZ         NOT NULL DEFAULT NOW(),
    updated_at              TIMESTAMPTZ         NOT NULL DEFAULT NOW(),
    deleted_at              TIMESTAMPTZ         NULL,

    CONSTRAINT pk_jobs PRIMARY KEY (id),

    CONSTRAINT fk_jobs_recruiter
        FOREIGN KEY (recruiter_id)
        REFERENCES recruiter_profiles (id)
        ON DELETE RESTRICT,

    CONSTRAINT fk_jobs_company
        FOREIGN KEY (company_id)
        REFERENCES companies (id)
        ON DELETE RESTRICT,

    CONSTRAINT fk_jobs_approved_by
        FOREIGN KEY (approved_by)
        REFERENCES users (id)
        ON DELETE RESTRICT,

    CONSTRAINT chk_jobs_min_cgpa
        CHECK (min_cgpa IS NULL OR (min_cgpa >= 0.00 AND min_cgpa <= 10.00)),

    CONSTRAINT chk_jobs_approval_consistency
        CHECK (
            (approved_by IS NULL AND approved_at IS NULL)
            OR
            (approved_by IS NOT NULL AND approved_at IS NOT NULL)
        ),

    CONSTRAINT chk_jobs_location_city
        CHECK (
            location_type = 'REMOTE'
            OR location_city IS NOT NULL
        )
);

COMMENT ON TABLE jobs IS 'Job postings created by recruiters, approved by admins, browsed by students.';
COMMENT ON COLUMN jobs.compensation IS 'Human-readable compensation string. Not a numeric to accommodate CTC and stipend formats.';
COMMENT ON COLUMN jobs.min_cgpa IS 'Minimum CGPA for eligibility. Application-layer enforced; NULL means no CGPA filter.';
COMMENT ON COLUMN jobs.closed_at IS 'Set when job is closed manually by recruiter or automatically when deadline passes.';
```

### 6.12 `applications`

**Purpose:** Records a student's application to a job using a specific resume. The composite UNIQUE constraint on `(student_id, job_id)` enforces the one-application-per-student-per-job rule at the database level (SRS APP-FR-002), making it impossible for duplicate applications to be created even under concurrent requests.

The `resume_id` FK deliberately uses `ON DELETE RESTRICT` — a resume that has been used in an application cannot be physically deleted, ensuring the application record always has a valid resume reference. Soft delete on resumes handles this scenario.

```sql
CREATE TABLE applications (
    id                  UUID                NOT NULL DEFAULT gen_random_uuid(),
    student_id          UUID                NOT NULL,
    job_id              UUID                NOT NULL,
    resume_id           UUID                NOT NULL,
    status              application_status  NOT NULL DEFAULT 'APPLIED',
    applied_at          TIMESTAMPTZ         NOT NULL DEFAULT NOW(),
    updated_at          TIMESTAMPTZ         NOT NULL DEFAULT NOW(),
    status_updated_at   TIMESTAMPTZ         NOT NULL DEFAULT NOW(),
    -- Tracks when the status last changed; separate from updated_at
    -- which can change for other reasons

    CONSTRAINT pk_applications PRIMARY KEY (id),

    CONSTRAINT fk_applications_student
        FOREIGN KEY (student_id)
        REFERENCES student_profiles (id)
        ON DELETE RESTRICT,

    CONSTRAINT fk_applications_job
        FOREIGN KEY (job_id)
        REFERENCES jobs (id)
        ON DELETE RESTRICT,

    CONSTRAINT fk_applications_resume
        FOREIGN KEY (resume_id)
        REFERENCES resumes (id)
        ON DELETE RESTRICT,
    -- Prevent resume deletion if used in any application

    CONSTRAINT uq_applications_student_job
        UNIQUE (student_id, job_id)
    -- Enforces one application per student per job (SRS APP-FR-002)
);

COMMENT ON TABLE applications IS 'Student applications to jobs. UNIQUE(student_id, job_id) prevents duplicate applications.';
COMMENT ON COLUMN applications.status_updated_at IS 'Timestamp of the last status transition. Separate from updated_at to support status age queries.';
```

### 6.15 `refresh_tokens`

**Purpose:** Stores hashed refresh tokens for JWT session management. The actual token value is never stored — only its SHA-256 hash. The `family_id` groups tokens that belong to the same login session. On token rotation (refresh), the old token is revoked and a new one is issued in the same family. If a revoked token is presented again, the entire family is invalidated (all rows with that `family_id`), protecting against refresh token theft.

```sql
CREATE TABLE refresh_tokens (
    id          UUID        NOT NULL DEFAULT gen_random_uuid(),
    user_id     UUID        NOT NULL,
    token_hash  VARCHAR(255) NOT NULL,
    -- SHA-256 hash of the raw token value. Raw token is never stored.
    family_id   UUID        NOT NULL,
    -- Groups tokens from the same login session for family invalidation
    expires_at  TIMESTAMPTZ NOT NULL,
    is_revoked  BOOLEAN     NOT NULL DEFAULT FALSE,
    revoked_at  TIMESTAMPTZ NULL,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),

    CONSTRAINT pk_refresh_tokens PRIMARY KEY (id),

    CONSTRAINT fk_refresh_tokens_user
        FOREIGN KEY (user_id)
        REFERENCES users (id)
        ON DELETE CASCADE,

    CONSTRAINT uq_refresh_tokens_token_hash
        UNIQUE (token_hash),

    CONSTRAINT chk_refresh_tokens_revoked_at
        CHECK (
            (is_revoked = FALSE AND revoked_at IS NULL)
            OR
            (is_revoked = TRUE  AND revoked_at IS NOT NULL)
        )
);

COMMENT ON TABLE refresh_tokens IS 'JWT refresh token store. Hashed values only. family_id enables reuse detection and session-family invalidation.';
COMMENT ON COLUMN refresh_tokens.token_hash IS 'SHA-256 hash of the raw opaque refresh token. The raw value is only held by the client.';
COMMENT ON COLUMN refresh_tokens.family_id IS 'UUID grouping all tokens from the same login session. Invalidating by family_id logs out all sessions sharing that origin.';
```

### 6.18 `audit_log`

**Purpose:** An immutable, append-only audit trail of all significant write operations in the system. Every `CREATE`, `UPDATE`, `DELETE`, and key authentication event produces a row. `old_values` and `new_values` are JSONB snapshots of the relevant entity fields at the time of the event.

This table has **no `updated_at` column** and **no soft delete column** because it must never be modified. Rows are written once and read many times. A future database-level row security policy (PostgreSQL RLS) can enforce immutability.

`actor_role` and `actor_email` are **denormalized** by design. If a user's role or email changes later, the audit log must still reflect what it was at the time of the event. Joining to `users` at read time would give the current value, not the historical value.

```sql
CREATE TABLE audit_log (
    id              UUID            NOT NULL DEFAULT gen_random_uuid(),
    entity_type     VARCHAR(100)    NOT NULL,
    -- e.g., 'USER', 'JOB', 'APPLICATION', 'COMPANY', 'RECRUITER'
    entity_id       UUID            NOT NULL,
    -- UUID of the affected entity
    action          audit_action    NOT NULL,
    actor_id        UUID            NULL,
    -- NULL for system-initiated actions (e.g., automated job expiry)
    actor_role      VARCHAR(50)     NULL,
    -- Denormalized: role at the time of the event
    actor_email     VARCHAR(255)    NULL,
    -- Denormalized: email at the time of the event
    old_values      JSONB           NULL,
    -- JSON snapshot of relevant fields before the change (NULL for CREATE)
    new_values      JSONB           NULL,
    -- JSON snapshot of relevant fields after the change (NULL for DELETE)
    ip_address      INET            NULL,
    -- Client IP extracted from X-Real-IP header by the application
    user_agent      VARCHAR(500)    NULL,
    created_at      TIMESTAMPTZ     NOT NULL DEFAULT NOW(),
    -- No updated_at. No deleted_at. This table is immutable.

    CONSTRAINT pk_audit_log PRIMARY KEY (id)
    -- Intentionally no FK on actor_id: the user may be deleted,
    -- but the audit record must survive and retain the denormalized fields.
);

COMMENT ON TABLE audit_log IS 'Immutable, append-only audit trail. No UPDATE or DELETE permitted. actor_role and actor_email are intentionally denormalized.';
COMMENT ON COLUMN audit_log.actor_id IS 'user_id of the actor. NULL for system actions. No FK constraint: audit records outlive user records.';
COMMENT ON COLUMN audit_log.old_values IS 'JSONB snapshot of relevant fields before the change. NULL for CREATE actions.';
COMMENT ON COLUMN audit_log.new_values IS 'JSONB snapshot of relevant fields after the change. NULL for DELETE actions.';
COMMENT ON COLUMN audit_log.ip_address IS 'Uses PostgreSQL INET type which validates IP format and supports CIDR range queries.';
```

**ON DELETE policy rationale:**
- `CASCADE` — Child records have no independent meaning without the parent (skills, education, tokens owned by a user).
- `RESTRICT` — The parent cannot be deleted while children exist; the application must handle the lifecycle (applications cannot be orphaned from jobs).
- `SET NULL` — The FK is nullable and the child can exist without the parent (recruiter without a company after company deactivation).

## 10. Soft Delete Strategy

### Philosophy

PlaceSync uses soft deletes on business entities where historical referential integrity is more important than physical removal. "Deleted" records remain in the database with a `deleted_at` timestamp set; they are excluded from all normal application queries using a `WHERE deleted_at IS NULL` clause.

Physical deletion is not implemented in V1 (SRS USER-FR-005).

### Tables Using Soft Delete

| Table | Soft Delete Column | Rationale |
|---|---|---|
| `users` | `deleted_at` | Deactivated users must retain their profile and application history. |
| `resumes` | `deleted_at` | Resumes referenced by existing applications cannot be physically removed. They are soft-deleted to prevent new applications from using them. |
| `jobs` | `deleted_at` | Admin-removed job postings retain their application and interview history for audit purposes. |
| `companies` | `deleted_at` | Deactivated companies retain their job and application history. |

### Tables Using Status-Based Lifecycle (Not Soft Delete)

| Table | Status Column | Reason for Status Pattern |
|---|---|---|
| `recruiter_profiles` | `verification_status` | Verification is a multi-state lifecycle, not a binary active/deleted state. |
| `applications` | `status` | Applications have a defined, ordered lifecycle. "Cancelled" is not a product concept in V1. |
| `interviews` | `status` | Interviews can be CANCELLED but the record is preserved for notification and audit history. |

### Tables with No Delete Mechanism

| Table | Reason |
|---|---|
| `notifications` | Marked as read, never deleted. Retention is a feature. |
| `audit_log` | Immutable by design. No delete ever. |
| `refresh_tokens` | Revoked in place; expired tokens cleaned up by a background job. |
| `email_verification_tokens` | Used flag set; expired tokens cleaned up by a background job. |
| `password_reset_tokens` | Used flag set; expired tokens cleaned up by a background job. |

### Query Pattern for Soft-Deleted Tables

All queries against soft-deletable tables must add the `deleted_at IS NULL` predicate. The partial indexes defined in Section 9 already embed this predicate, making these queries efficient.

```sql
-- Standard active record query pattern
SELECT * FROM users
WHERE deleted_at IS NULL
AND email = $1;

-- Active jobs only
SELECT * FROM jobs
WHERE status = 'OPEN'
AND deleted_at IS NULL
AND application_deadline > NOW();

-- Performing a soft delete
UPDATE users
SET deleted_at = NOW(),
    updated_at = NOW(),
    is_active  = FALSE
WHERE id = $1;
```

### Cascading Soft Deletes

Soft deleting a `user` does **not** automatically soft-delete their child records (student_profiles, recruiter_profiles, etc.). The application service layer is responsible for deciding what cascade logic is appropriate. In V1, soft-deleting a user:

1. Sets `users.deleted_at = NOW()` and `users.is_active = FALSE`.
2. The child profile remains in place (historical job/application data is preserved).
3. The user can no longer log in (checked against `is_active`).

## 11. Auditing Strategy

### Approach

PlaceSync implements **application-level audit logging** via a Spring AOP `@Aspect`. The aspect intercepts service methods annotated with `@Audited` (or all write-path service methods), captures the before/after state of the entity, and inserts a row into `audit_log` within the same database transaction.

### What Gets Audited

| Event Category | Entities | Action Values |
|---|---|---|
| Entity writes | users, student_profiles, recruiter_profiles, companies, jobs, applications, interviews | CREATE, UPDATE, SOFT_DELETE |
| Authentication | users | LOGIN_SUCCESS, LOGIN_FAILURE, LOGOUT, PASSWORD_CHANGE, PASSWORD_RESET, EMAIL_VERIFIED |
| Admin actions | users, recruiter_profiles, companies, jobs | UPDATE (verification decisions are UPDATE events on the relevant entity) |
| Account lifecycle | users | ACCOUNT_LOCKED, ACCOUNT_UNLOCKED |

### What Is NOT Audited

- Read operations (SELECT). The audit log is a write audit trail.
- Notification creation (too high volume; use application logs instead).
- Token issuance and expiry (tracked in their own tables).
- Child collection changes (student_skills, student_educations) are low-risk and not individually audited in V1.

### Audit Log Entry Structure

Each row captures:

```
entity_type   → which domain entity was affected (e.g., 'JOB')
entity_id     → UUID of that specific entity
action        → what happened (e.g., 'UPDATE')
actor_id      → who did it (the authenticated user's UUID)
actor_role    → their role at that moment (denormalized)
actor_email   → their email at that moment (denormalized)
old_values    → JSONB of significant fields before change (null for CREATE)
new_values    → JSONB of significant fields after change (null for DELETE)
ip_address    → client IP from X-Real-IP header
user_agent    → client User-Agent string
created_at    → exact UTC timestamp
```

### JSONB Payload Conventions

`old_values` and `new_values` capture only **significant fields** — not every column. Sensitive fields are explicitly excluded.

**Never included in audit payloads:**
- `password_hash`
- `token_hash` (any token table)
- `oauth_provider_id`

**Example — Job status change (OPEN → CLOSED):**
```json
{
  "old_values": {
    "status": "OPEN",
    "closed_at": null
  },
  "new_values": {
    "status": "CLOSED",
    "closed_at": "2026-06-23T10:00:00Z"
  }
}
```

**Example — Application status change:**
```json
{
  "old_values": { "status": "UNDER_REVIEW" },
  "new_values": { "status": "SHORTLISTED" }
}
```

### Immutability Enforcement

In V1, immutability is enforced by convention: no UPDATE or DELETE statement against `audit_log` exists anywhere in the application codebase. In V2, PostgreSQL Row-Level Security (RLS) can enforce this at the database level:

```sql
-- V2 — enforce audit log immutability with RLS
ALTER TABLE audit_log ENABLE ROW LEVEL SECURITY;

CREATE POLICY audit_log_insert_only
    ON audit_log
    FOR ALL
    USING (FALSE)       -- No SELECT/UPDATE/DELETE via policy
    WITH CHECK (TRUE);  -- INSERT allowed

-- Override for the application role only
GRANT INSERT ON audit_log TO placesync_app;
```

### Retention

Audit logs are retained indefinitely in V1. The SRS (NFR-062) requires minimum 12-month retention. In V2, a partitioned table by `created_at` month can enable efficient archival of old partitions to cold storage.

## 12. Shared Trigger Function

All tables with an `updated_at` column use a shared PostgreSQL trigger function to automatically set `updated_at = NOW()` on every UPDATE. This removes any dependency on the application layer remembering to set this field.

```sql
-- Created in V001 migration, before any table that uses it

CREATE OR REPLACE FUNCTION set_updated_at()
RETURNS TRIGGER
LANGUAGE plpgsql
AS $$
BEGIN
    NEW.updated_at = NOW();
    RETURN NEW;
END;
$$;
```

**Applied to each table:**

```sql
CREATE TRIGGER trg_users_updated_at
    BEFORE UPDATE ON users
    FOR EACH ROW EXECUTE FUNCTION set_updated_at();

CREATE TRIGGER trg_student_profiles_updated_at
    BEFORE UPDATE ON student_profiles
    FOR EACH ROW EXECUTE FUNCTION set_updated_at();

CREATE TRIGGER trg_student_educations_updated_at
    BEFORE UPDATE ON student_educations
    FOR EACH ROW EXECUTE FUNCTION set_updated_at();

CREATE TRIGGER trg_student_experiences_updated_at
    BEFORE UPDATE ON student_experiences
    FOR EACH ROW EXECUTE FUNCTION set_updated_at();

CREATE TRIGGER trg_companies_updated_at
    BEFORE UPDATE ON companies
    FOR EACH ROW EXECUTE FUNCTION set_updated_at();

CREATE TRIGGER trg_recruiter_profiles_updated_at
    BEFORE UPDATE ON recruiter_profiles
    FOR EACH ROW EXECUTE FUNCTION set_updated_at();

CREATE TRIGGER trg_jobs_updated_at
    BEFORE UPDATE ON jobs
    FOR EACH ROW EXECUTE FUNCTION set_updated_at();

CREATE TRIGGER trg_applications_updated_at
    BEFORE UPDATE ON applications
    FOR EACH ROW EXECUTE FUNCTION set_updated_at();

CREATE TRIGGER trg_interviews_updated_at
    BEFORE UPDATE ON interviews
```

`audit_log`, `notifications`, and the token tables do not have `updated_at` and therefore do not have triggers.

## 13. Flyway Migration Ordering

All migration scripts live in `src/main/resources/db/migration/`. Flyway applies them in version order on application startup. The naming convention is `V{version}__{description}.sql`.

**Rule:** Version numbers are sequential integers. A migration must never be modified after it has been applied to any environment. Corrections require a new migration script.

### Migration Sequence

| `V001__create_enum_types.sql` | none | All custom `CREATE TYPE` statements. All ENUM types and the `set_updated_at()` trigger function. |

### Migration Authoring Rules

1. **Never modify** a script that has been applied to staging or production. Create a new `V{n}` script instead.
2. Scripts must be **idempotent where possible** — use `CREATE TABLE IF NOT EXISTS` and `CREATE INDEX IF NOT EXISTS`.
3. Each script must run as a **single transaction** so that a partial failure leaves the schema unchanged.
4. **Test migrations on a copy of the production schema** before applying to production.
5. **Destructive migrations** (DROP, ALTER COLUMN TYPE, DELETE) require a two-step approach: deprecate in one release, remove in the next.
